VSTUPNÍ TESTOVACÍ SCÉNÁŘ
================================================================================

TS1 Vyhledání a stažení LV podle čísla LV a katastrálního území stavby

Cíl scénáře:
Ověřit základní funkčnost vyhledání a stažení listu vlastnictví (LV) při zadání povinných kritérií.

Předpoklady:
Uživatel je přihlášen do aplikace a jsou k dispozici testovací hodnoty pro katastrální území a číslo LV.

Postup:
1. Otevřít záložku Vyhledávání LV a zvolit vyhledávání podle čísla LV.
2. Vyplnit pole Katastrální území a Číslo LV.
3. Spustit akci Stáhnout LV.

Očekávaný výsledek:
Aplikace přijme požadavek a zobrazí výsledek stažení LV. Pokud jsou vstupy neúplné nebo neplatné, aplikace zobrazí validaci a nedojde k zahájení stažení.

Vytvoř mi jeden happy flow testovací scénář pro aplikaci KN Proxy na stránce:
https://knproxy-prs.csint.cz/knproxy/

FINÁLNÍ GITHUB ACTIONS WORKFLOW
================================================================================
name: Robot Framework CI

on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
  workflow_dispatch:

jobs:
  robot-tests:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'
          cache: 'pip'

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install robotframework robotframework-browser

      - name: Initialize Robot Framework Browser
        run: rfbrowser init

      - name: Run Robot Framework tests
        run: |
          robot --outputdir results --variable HEADLESS:True .
        continue-on-error: true

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: robot-test-results
          path: results/
          retention-days: 14

### 3. Zhodnocení pokrytí
*   **Happy Flow:** Testovací artefakty (Gherkin i Robot Framework) plně pokrývají pozitivní cestu (Happy Flow) definovanou v postupu původního scénáře.
*   **Struktura:** Reviewed Robot Framework test výrazně zlepšuje robustnost testu přidáním explicitních čekání (`Wait For Elements State`) a rozdělením kroků pro lepší čitelnost a znovupoužitelnost.
*   **Technické parametry:** Jsou definovány proměnné pro URL i testovací data, což usnadňuje údržbu.

### 4. Případná rizika nebo mezery v pokrytí
*   **Chybějící negativní testy (R8):** Původní scénář v sekci "Očekávaný výsledek" zmiňuje chování aplikace při neúplných nebo neplatných vstupech. Aktuální testy se zaměřují pouze na úspěšné stažení. Pro 100% pokrytí celého zadání by bylo nutné doplnit scénáře pro validaci polí.
*   **Hardcoded data:** Testovací data (KU_VALUE, LV_VALUE) jsou definována přímo v testu. Pro rozsáhlejší testování by bylo vhodné využít externí datový soubor.
*   **URL adresa:** V Robot Framework testech je použita placeholder adresa `https://knproxy.example.com` namísto adresy uvedené v zadání (`https://knproxy-prs.csint.cz/knproxy/`).

### 5. Krátké shrnutí
Vytvořené artefakty **plně pokrývají požadovaný Happy Flow** scénář TS1. Test je logicky strukturován a připraven k automatizaci v nástroji Robot Framework s využitím knihovny Browser (Playwright). Pro kompletní pokrytí všech aspektů původního zadání doporučuji v dalším kroku vytvořit negativní testovací scénáře zaměřené na validaci vstupních polí.

Následná revize Robot Framework testu zvýšila technickou úroveň skriptu. Byla implementována explicitní čekání na stav elementů, což zvyšuje odolnost testu proti nestabilitě síťové odezvy.
