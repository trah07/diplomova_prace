VSTUPNÍ TESTOVACÍ SCÉNÁŘ
================================================================================

TS1 Vyhledání a stažení LV podle čísla LV a katastrálního území stavby

Cíl scénáře:
Ověřit základní funkčnost vyhledání a stažení listu vlastnictví (LV) při zadání povinných kritérií.

Předpoklady:
Uživatel je přihlášen do aplikace a jsou k dispozici testovací hodnoty pro katastrální území a číslo LV.

Postup:
1. Otevřít záložku Vyhledávání LV a zvolit vyhledávání podle čísla LV.
2. Vyplnit pole Katastrální území a Číslo LV.
3. Spustit akci Stáhnout LV.

Očekávaný výsledek:
Aplikace přijme požadavek a zobrazí výsledek stažení LV. Pokud jsou vstupy neúplné nebo neplatné, aplikace zobrazí validaci a nedojde k zahájení stažení.

Vytvoř mi jeden happy flow testovací scénář pro aplikaci KN Proxy na stránce:
https://knproxy-prs.csint.cz/knproxy/

FINÁLNÍ GITHUB ACTIONS WORKFLOW
================================================================================
name: Robot Framework CI

on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
  workflow_dispatch:

jobs:
  robot-tests:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'

      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install Python dependencies
        run: |
          python -m pip install --upgrade pip
          pip install robotframework robotframework-browser

      - name: Initialize Robot Framework Browser
        run: rfbrowser init

      - name: Run Robot Framework tests
        run: robot --outputdir results .

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: robot-framework-report
          path: results/
          retention-days: 30

- **P7 (Výsledek - Negativní):** Zobrazení validace při neplatných vstupech (v rámci Happy Flow scénáře není vyžadováno, ale je součástí TS1).

### 3. Zhodnocení pokrytí
- **Gherkin Scenario:** Přesně reflektuje kroky definované v TS1 pro pozitivní průchod. Struktura Given-When-Then je logická a srozumitelná.
- **Robot Framework Draft:** Implementuje všechny kroky z Gherkinu. Používá konkrétní testovací data (Adamov, LV 115), čímž naplňuje předpoklad o dostupnosti testovacích hodnot.
- **Reviewed Robot Framework Test:** Zvyšuje kvalitu pokrytí přidáním robustních prvků (čekání na stavy elementů `Wait For Elements State`, použití proměnných `${KU_NAME}`, `${LV_NUMBER}`). Tento test plně pokrývá funkční požadavky P1–P6 v automatizované podobě.

---

### 4. Případná rizika nebo mezery v pokrytí
1. **Ověření staženého souboru:** Test končí ověřením viditelnosti úspěšné zprávy (`.alert-success`). Původní scénář zmiňuje "výsledek stažení LV". V automatizaci by bylo ideální doplnit kontrolu, zda byl soubor skutečně stažen do souborového systému, pokud je to cílem testu.
2. **Negativní scénáře:** Původní TS1 zmiňuje chování při neúplných vstupech. Tyto stavy nejsou v aktuálním Happy Flow pokryty. Pro plné pokrytí TS1 by bylo nutné vytvořit další testovací případy (např. chybějící číslo LV).
3. **Závislost na datech:** Test využívá konkrétní katastrální území a číslo LV. Pokud by tato data v testovacím prostředí zanikla, test selže, i když aplikace bude funkční.

---

### 5. Krátké shrnutí
Vygenerované testovací artefakty **plně pokrývají happy flow** definovaný v původním scénáři TS1. Přechod z textového zadání přes Gherkin až po revidovaný Robot Framework kód proběhl bez ztráty kontextu. Revidovaný test je stabilní, obsahuje nezbytné validace a je připraven k nasazení do CI/CD prostředí pro ověření základní funkčnosti vyhledávání LV.

FINÁLNÍ REPORT
================================================================================
Tato technická zpráva shrnuje proces transformace manuálního testovacího zadání do podoby plně automatizovaného testovacího cyklu pro aplikaci KN Proxy.

**Vstupní testovací scénář**
Výchozím bodem byl scénář TS1, který definoval postup pro vyhledání a stažení listu vlastnictví (LV) na základě čísla LV a katastrálního území. Scénář specifikoval nezbytné předpoklady, kroky postupu a očekávané chování aplikace pro tzv. „happy flow“.

**Gherkin scénář**
Prostřednictvím Gherkin syntaxe byl původní text formalizován do strukturovaného formátu Given-When-Then. Tento krok zajistil srozumitelné propojení mezi obchodními požadavky a technickou implementací, přičemž přesně definoval uživatelské interakce a očekávané stavy systému.

**Návrh a revize Robot Framework testu**
Prvotní technický návrh (draft) v nástroji Robot Framework byl následně podroben expertní revizi. Výsledný revidovaný test využívá moderní knihovnu Browser (Playwright), implementuje parametrizaci pomocí proměnných a obsahuje robustní mechanismy pro asynchronní čekání na stav elementů. Tím je zajištěna vysoká stabilita testu i v dynamickém webovém prostředí.

**Pracovní postup GitHub Actions**
Pro účely kontinuální integrace (CI) byl vytvořen konfigurační soubor pro GitHub Actions. Tento automatizovaný workflow zajišťuje přípravu běhového prostředí (Python, Node.js, závislosti), inicializaci prohlížečů a samotné spuštění testů s následným exportem výsledků ve formě artefaktů.

**Analýza pokrytí a role lidské validace**
Analýza pokrytí potvrdila, že automatizovaný test plně reflektuje všechny funkční požadavky definované v původním zadání pro pozitivní průchod scénářem. Lidská validace v tomto procesu sehrála kritickou roli při optimalizaci selektorů, nastavení časových limitů a ověření logické správnosti celého řetězce, což eliminuje riziko falešně negativních výsledků.

**Celkový účel workflow**
Hlavním účelem tohoto workflow je vytvoření udržitelného a opakovatelného procesu pro zajištění kvality softwaru. Automatizace převodu textového zadání do spustitelného kódu v rámci CI/CD infrastruktury umožňuje rychlou detekci regresních chyb a zvyšuje efektivitu vývojového cyklu aplikace KN Proxy.
