VSTUPNÍ TESTOVACÍ SCÉNÁŘ

TS2 Vyhledání a stažení LV podle nemovitosti Cíl: Ověřit funkčnost vyhledání LV

Cíl: Ověřit funkčnost vyhledání LV prostřednictvím záložky „podle nemovitosti".

Předpoklady: Jsou k dispozici platné identifikační údaje nemovitosti.

Postup:
1. Otevřít záložku „Vyhledávání LV" a zvolit podzáložku „podle nemovitosti".
2. Vyplnit povinný pole „Typ nemovitosti“, „Katastrální území“ a „Kmenové číslo“.
3. Spustit vyhledání a stažení LV.
4. Ověřit zobrazení výsledku a existenci staženého souboru.

Očekávaný výsledek: Aplikace vrátí LV odpovídající zadané nemovitosti.

Zadání: Vytvoř mi jeden testovací scénář pro aplikaci kn proxy na stránce https://knproxy-prs.csint.cz/knproxy/.

name: Robot Framework CI

on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
  workflow_dispatch:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.10'
          cache: 'pip'

      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install robotframework robotframework-browser

      - name: Install Browser binaries and dependencies
        run: |
          rfbrowser init
          # Ensure system dependencies for Playwright are installed on the runner
          npx playwright install-deps chromium

      - name: Run Robot Framework tests
        run: |
          robot --outputdir results .
        continue-on-error: true

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: robot-test-results
          path: |
            results/report.html
            results/log.html
            results/output.xml
            results/browser/screenshot/
          retention-days: 5

Tato technická zpráva shrnuje proces transformace manuálního testovacího zadání do plně automatizované podoby v rámci workflow generování testů pro aplikaci KN Proxy.

**Vstupní testovací scénář**
Výchozím bodem byl scénář TS2 zaměřený na vyhledání a stažení listu vlastnictví (LV) prostřednictvím vyhledávání podle nemovitosti. Scénář definoval kroky pro navigaci v aplikaci, vyplnění povinných polí (typ nemovitosti, katastrální území, kmenové číslo) a následné ověření výsledku včetně stažení souboru.

**Generovaný Gherkin scénář**
Pro zajištění srozumitelnosti a strukturovanosti byl manuální postup převeden do formátu Gherkin. Tento zápis využívá deklarativní styl Given-When-Then, který slouží jako most mezi byznysovým zadáním a technickou implementací, přičemž přesně mapuje všechny kroky původního postupu.

**Návrh a revize Robot Framework testu**
Prvotní technický návrh (draft) v nástroji Robot Framework definoval základní interakci s prohlížečem pomocí knihovny Browser. Následná revize tento návrh povýšila na produkční úroveň. Revidovaný test obsahuje robustní mechanismy čekání na prvky, parametrizaci vstupních dat pomocí proměnných a pokročilé zpracování stahování souborů. Pro fyzické ověření existence souboru na disku byla integrována knihovna OperatingSystem.

**Pracovní postup GitHub Actions**
Pro zajištění kontinuální integrace byl vytvořen konfigurační soubor pro GitHub Actions. Tento workflow automatizuje celý proces testování: od přípravy prostředí (instalace Pythonu a prohlížečů Playwright) přes samotné spuštění testů až po archivaci výsledků a logů ve formě artefaktů.

**Analýza pokrytí a role lidské validace**
Analýza potvrdila, že automatizované testy stoprocentně pokrývají definované požadavky (navigace, vstupní data, akce, UI verifikace a správa souborů). Lidská validace byla v procesu nezbytná pro identifikaci rizik spojených se statickými daty a pro doplnění logických kontrol, které zvyšují stabilitu testu v dynamickém webovém prostředí.

**Celkový účel workflow**
Hlavním cílem tohoto workflow je vytvoření spolehlivého, opakovatelného a udržovatelného mechanismu pro ověřování klíčových funkcí aplikace. Výsledkem je plně automatizovaný řetězec, který minimalizuje potřebu manuálního testování a umožňuje rychlou detekci chyb v rámci vývojového cyklu.
