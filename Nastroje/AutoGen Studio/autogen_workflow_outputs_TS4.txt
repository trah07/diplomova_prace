Jako senior inženýr automatizace jsem provedl revizi tvého Robot Framework kódu. Tvůj návrh byl dobrým základem, ale pro reálné a stabilní spuštění v knihovně Browser (Playwright) vyžadoval několik technických úprav:
Selektory: Upravil jsem selektory tak, aby byly robustnější (použití CSS a textových filtrů).
Práce s našeptávačem: Pole "Katastrální území" v těchto typech aplikací bývá často našeptávač (autocomplete). Doplnil jsem logiku pro vyčkání na našeptávač a potvrzení výběru.
Logika stahování: V knihovně Browser se stahování řeší asynchronně. Pokud očekáváme, že se soubor nestáhne, musíme to ošetřit tak, aby test neselhal na timeoutu při čekání na soubor, který nemá přijít.
Lokalizace chyb: Upřesnil jsem kontrolu chybové hlášky.

```
${state}=    Wait For Elements State    ${ERROR_MESSAGE_LOC}    visible    timeout=5s

# Využijeme Promise pro ověření, že download nebyl vyvolán
# Pokud do 3 sekund nezačne download, považujeme to za splněné

# Pokud download nezačne, Wait For selže na timeoutu, což je v tomto případě v pořádku
# Použijeme Run Keyword And Expect Error, abychom potvrdili, že soubor NEDORAZIL
Run Keyword And Expect Error    *Timeout*    Wait For    ${download_promise}
Log    Potvrzeno: Žádný soubor se nezačal stahovat.
```

Provedené úpravy a vysvětlení:
Wait For Elements State: Přidal jsem explicitní čekání na zobrazení formuláře po kliknutí na podzáložku. Je to nutné pro stabilitu v CI/CD.
Logika stahování (Promise): Wait For Download je v Playwrightu asynchronní. Tím, že jsem ho zabalil do Run Keyword And Expect Error, test projde pouze tehdy, když ke stažení nedojde (vyprší časový limit).
Suite Teardown: Přidal jsem Close Browser ALL, aby po testu nezůstávaly viset instance prohlížeče.
Selektory: Upravil jsem je na ID (input#katastralniUzemi), což je v automatizaci preferovaný způsob před hledáním podle jména. Poznámka: Pokud ID v aplikaci neexistují, lze se vrátit k input[name="..."].

AGENT_GITHUB_ACTIONS
================================================================================

Zde je kompletní GitHub Actions workflow YAML soubor připravený pro vaše testy. Tento soubor zajistí automatické spuštění testů při každém pushi nebo pull requestu do větve main.
Soubor uložte do svého repozitáře jako: .github/workflows/robot-tests.yml
```
name: Robot Framework Browser Tests
# Trigger: Spuštění při pushi nebo PR do větve main
on:
push:
branches: [ main ]
pull_request:
branches: [ main ]
jobs:
```

test-automation:
name: Run Robot Framework Tests
runs-on: ubuntu-latest
steps:
- name: Checkout Repository
uses: actions/checkout@v4
- name: Set up Python
uses: actions/setup-python@v5
with:
python-version: '3.10'
cache: 'pip'
- name: Install Dependencies
run: |
python -m pip install --upgrade pip
pip install robotframework robotframework-browser
- name: Initialize Browser Library
run: |
# Instalace prohlížečů a jejich systémových závislostí
rfbrowser init
- name: Run Robot Framework Tests
run: |
# Vytvoření složky pro výsledky a spuštění testů
# Předpokládá, že testy jsou ve složce tests/
mkdir -p results
robot --outputdir results tests/
# continue-on-error: true zajistí, že se upload provedou i při selhání testů
continue-on-error: true
- name: Upload Test Results
if: always()
uses: actions/upload-artifact@v4
with:
name: robot-test-results
path: results/
if-no-files-found: warn
retention-days: 7
Klíčové vlastnosti tohoto workflow:
Prostředí: Používá nejnovější Ubuntu a Python 3.10.
Inicializace: Příkaz rfbrowser init v rámci GitHub Actions automaticky nainstaluje potřebné binární soubory prohlížečů (Playwright).
Výsledky (Artifacts): Krok if: always() zajišťuje, že se logy, reporty a screenshoty (pokud vzniknou) nahrají do GitHubu i v případě, že testy selžou.
Struktura: Očekává, že vaše .robot soubory jsou umístěny v adresáři tests/.
Prosím o tvou validaci (OK/NEOK). Pokud je vše v pořádku, workflow je připraveno k nasazení.
